g_cpc_current has: Current CPC classification of the patent
CPC = Cooperative Patent Classification
It’s a system used to categorize patents by technology or field.
In patent classification, green tech is mostly captured using:
👉 CPC Section: Y
Y02 → Climate change mitigation technologies

Think of it like:
A library classification system (like Dewey Decimal)
But for inventions instead of books

Subcategories of Y02 
| Code | Meaning                               |
| ---- | ------------------------------------- |
| Y02E | Energy generation (solar, wind)       |
| Y02T | Transport (electric vehicles)         |
| Y02B | Buildings (energy efficiency)         |
| Y02P | Industry (low-emission manufacturing) |
| Y02C | Carbon capture                        |
| Y02W | Waste management                      |

The full Y02 universe answers the complete question: 
how is technology responding to climate change across every sector of the economy.

Y02 is architecturally designed to always be additional — never inventional. This is not an accident or an error. It is deliberate USPTO policy.
The reasoning is this: 
nobody invents something and says "I am inventing green technology." 
They invent a specific thing — a better sintering process, a new wind turbine blade mount, a more efficient battery chemistry. 
The Y02 tag is then applied by the examiner as a cross-cutting label that says "this invention has climate relevance."

A Y02T-focused study tells a clear, compelling story: how is technology transforming the way humanity moves?

PROJECT DEEP DIVE: Green Transport Patent Intelligence Pipeline
═══════════════════════════════════════════════════════════════════════════════

OVERVIEW
--------
This project ingests USPTO patent data (141,124 green transport patents from 1976–2025),
cleans and standardizes it, loads it into SQLite, analyzes it for insights, and visualizes
it through an interactive Streamlit dashboard. The focus is on CPC Y02T (climate change
mitigation in transportation).

═══════════════════════════════════════════════════════════════════════════════
PIPELINE ARCHITECTURE & CODE EXPLANATIONS
═══════════════════════════════════════════════════════════════════════════════

1. RAW DATA INGESTION (data/raw/*.tsv)
   ────────────────────────────────────
   Input Files:
   • g_patent.tsv - Base patent records (ID, title, date, type)
   • g_cpc_current.tsv - Patent classifications (links patents to Y02T category)
   • g_patent_abstract.tsv - Detailed summaries of what each patent does
   • g_inventor_disambiguated.tsv - Cleaned inventor identities + locations
   • g_assignee_disambiguated.tsv - Cleaned company/owner identities + locations
   • g_location_disambiguated.tsv - Standardized geographic locations

   Purpose:
   These 6 TSV files form a relational network. Each table connects to others
   via patent_id, inventor_id, or assignee_id. Raw data contains duplicates,
   inconsistencies, and non-UTF8 characters.

2. CLEANING STAGE (scripts/02_clean.py)
   ────────────────────────────────────
   
   Code Flow:
   a) Load all TSV files into pandas DataFrames
      df_patent = pd.read_csv('data/raw/g_patent.tsv', sep='\t', on_bad_lines='skip')
      └─ Explanation: on_bad_lines='skip' handles corrupted rows gracefully
      
   b) Filter to Y02T (green transport) patents only
      y02t_patents = df_cpc[df_cpc['cpc_section'] == 'Y02T']['patent_id']
      └─ Relevance: Isolates only climate-relevant transport tech from 10M+ patents
      
   c) Standardize text encoding & remove nulls
      df_patent['title'] = df_patent['title'].fillna('').str.replace(r'[^\x00-\x7F]', '', regex=True)
      └─ Relevance: Ensures database compatibility & consistent searching
      
   d) Validate dates are within reasonable range (1976–2025)
      df_patent = df_patent[(df_patent['year'] >= 1976) & (df_patent['year'] <= 2025)]
      └─ Relevance: Removes data entry errors (patents from year 0 or 3000)
      
   e) Remove duplicates by patent_id
      df_patent = df_patent.drop_duplicates(subset=['patent_id'])
      └─ Relevance: Same patent shouldn't appear twice in analysis
      
   f) Output cleaned data to data/clean/*.tsv
      df_patent.to_csv('data/clean/patent_clean.tsv', sep='\t', index=False)
      └─ Relevance: Creates audit trail; raw data untouched

3. DATA QUALITY CHECK (scripts/02b_data_quality.py)
   ────────────────────────────────────────────────
   
   Validations performed:
   a) Check for null values in critical columns
      if df_patent['patent_id'].isnull().sum() > 0:
          raise ValueError("Found nulls in patent_id")
      └─ Relevance: NULL IDs would break database relationships
      
   b) Verify referential integrity
      invalid_patents = inventors['patent_id'].isin(patent['patent_id']).sum() == 0
      └─ Relevance: Every inventor record must link to a valid patent
      
   c) Check for data type mismatches
      if not pd.api.types.is_numeric_dtype(df['year']):
          df['year'] = pd.to_numeric(df['year'], errors='coerce')
      └─ Relevance: Year must be int for sorting/filtering
      
   d) Validate geographic locations (country codes)
      valid_countries = ['US', 'JP', 'DE', 'FR', ...]
      df_inventor = df_inventor[df_inventor['country'].isin(valid_countries)]
      └─ Relevance: Invalid country codes indicate data corruption
      
   e) Generate quality report
      print(f"Total patents: {len(df_patent)}")
      print(f"Date range: {df_patent['year'].min()} – {df_patent['year'].max()}")
      └─ Relevance: Confirms pipeline correctness before database load

4. DATABASE LOAD (scripts/03_load_db.py)
   ───────────────────────────────────────
   
   Code Flow:
   a) Create SQLite schema from sql/schema.sql
      with sqlite3.connect('patents.db') as conn:
          conn.executescript(open('sql/schema.sql').read())
      └─ Relevance: Defines tables, indexes, and relationships upfront
      
   b) Create database constraints
      CREATE TABLE patents (
          patent_id TEXT PRIMARY KEY,
          title TEXT NOT NULL,
          year INT CHECK (year >= 1976 AND year <= 2025),
          FOREIGN KEY (cpc_id) REFERENCES cpc(cpc_id)
      );
      └─ Relevance: Ensures data integrity at database layer
      
   c) Batch insert cleaned data
      for chunk in pd.read_csv('data/clean/patent_clean.tsv', chunksize=5000, sep='\t'):
          chunk.to_sql('patents', conn, if_exists='append', index=False)
      └─ Relevance: Chunking prevents memory overflow on large datasets
      
   d) Create indexes for fast querying
      CREATE INDEX idx_patent_year ON patents(year);
      CREATE INDEX idx_inventor_country ON inventors(country);
      └─ Relevance: Dashboard queries run in <1 second with indexes

5. ANALYSIS STAGE (scripts/04_analyze.py)
   ──────────────────────────────────────
   
   SQL Queries Used:
   
   a) Patent growth over time
      SELECT year, COUNT(*) as patents FROM patents
      WHERE year >= 1976 GROUP BY year ORDER BY year;
      └─ Relevance: Identifies inflection points in green tech adoption
      
   b) Top inventors by patent count
      SELECT i.name, COUNT(DISTINCT r.patent_id) as patent_count
      FROM inventors i JOIN relations r ON i.inventor_id = r.inventor_id
      GROUP BY i.inventor_id ORDER BY patent_count DESC LIMIT 20;
      └─ Relevance: Identifies prolific innovators in green transport
      
   c) Technology distribution by country
      SELECT l.country, COUNT(DISTINCT r.patent_id) as patents
      FROM locations l JOIN inventors i ON l.location_id = i.location_id
      JOIN relations r ON i.inventor_id = r.inventor_id
      GROUP BY l.country ORDER BY patents DESC;
      └─ Relevance: Shows which countries lead in green transport R&D
      
   d) Company portfolio analysis
      SELECT c.name, COUNT(DISTINCT r.patent_id) as patents,
             MIN(p.year) as first_patent, MAX(p.year) as latest_patent
      FROM companies c JOIN relations r ON c.company_id = r.company_id
      JOIN patents p ON r.patent_id = p.patent_id
      GROUP BY c.company_id ORDER BY patents DESC;
      └─ Relevance: Identifies market leaders and entry points
